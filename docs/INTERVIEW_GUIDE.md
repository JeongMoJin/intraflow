# Interview Guide

## 왜 ASP.NET Core를 사용했는가?

사내 인트라넷은 인증, 권한, REST API, DB 연동, Swagger 문서화가 중요합니다. ASP.NET Core는 C# 기반으로 이런 기능을 안정적으로 구성할 수 있고, 공고의 .NET 8 요구사항과도 직접 맞기 때문에 선택했습니다.

## EF Core 사용 시 장단점은?

장점은 LINQ로 생산성 있게 DB 접근을 할 수 있고 provider 전환이 비교적 쉽다는 점입니다. 단점은 쿼리 성능이나 N+1 문제가 숨어 있을 수 있어서 실제 운영에서는 SQL 로그와 실행 계획을 같이 봐야 합니다.

## ERP 연동에서 가장 중요한 점은?

데이터 매핑과 실패 처리라고 생각합니다. 외부 ERP의 ID와 내부 ID를 매핑하고, 동기화 실패 시 재처리할 수 있게 로그를 남겨야 운영 중 원인을 추적할 수 있습니다.

## 감사 로그를 왜 남겨야 하는가?

사내 시스템에서는 누가 언제 어떤 정보를 변경했는지 추적할 수 있어야 합니다. 특히 결재 승인/반려, 직원 정보 수정, ERP Sync 같은 작업은 장애나 보안 이슈가 생겼을 때 감사 로그가 근거가 됩니다.

## Microsoft Graph 연동은 어떻게 확장할 수 있는가?

현재는 `IGraphUserService` 인터페이스와 Mock 구현을 사용했습니다. 실제 연동 시에는 RealGraphUserService에 Microsoft Graph SDK를 연결하고, Entra ID 앱 등록의 tenant/client 정보를 secret store에서 주입하면 됩니다.

## Entra ID SSO를 실제로 붙인다면 어떤 흐름인가?

프론트에서 Entra ID로 로그인하고, API는 발급된 access token을 검증합니다. 사용자 claim에서 email과 role 또는 group 정보를 읽고, 내부 AppUser/Employee와 매핑해 권한을 적용하는 흐름으로 확장할 수 있습니다.

## AI 코딩 도구를 어떻게 활용했는가?

요구사항 분해, 코드 초안 작성, 문서 초안 작성에 활용했습니다. 다만 그대로 신뢰하지 않고 빌드, 테스트, 패키지 경고, 권한 흐름, secret 포함 여부를 직접 확인했습니다.

## AI 결과물을 어떻게 검증했는가?

`dotnet build`, `dotnet test`, `npm run build`를 실행했고, 권한별 메뉴, 이메일 마스킹, AuditLog 저장, CSV import 실패 로그 같은 핵심 시나리오를 테스트 대상으로 잡았습니다.

## 보안상 주의한 점은?

실제 secret을 넣지 않았고, appsettings.example에는 placeholder만 넣었습니다. Manager 권한에서는 이메일을 마스킹하고, AuditLog에는 비밀번호나 client secret 형태의 값이 남지 않도록 처리했습니다.

## 레거시 시스템을 고도화할 때 주의할 점은?

한 번에 모든 것을 바꾸기보다 CSV import처럼 작은 경로를 먼저 만들고, 실패 데이터를 로그로 남기면서 점진적으로 신뢰도를 높이는 방식이 좋다고 생각합니다. 기존 데이터 품질을 먼저 확인하는 것도 중요합니다.

## 경력연수가 공고보다 조금 부족한데 어떻게 보완할 수 있는가?

아직 모든 운영 상황을 경험했다고 말하기보다는, 실무 요구를 빠르게 이해하고 작은 범위에서 끝까지 구현/검증하는 능력을 보여주고 싶습니다. 이 프로젝트도 권한, 감사 로그, ERP 연계, 문서화처럼 실제 업무에서 중요한 지점을 직접 구현해 본 사례로 설명할 수 있습니다.
